## M-3 — Reproduce on non-synthetic data

### Why

Today's two experiments used curated synthetic corpora. Their findings are
classified `[T2] [CAVEAT: single agent, curated corpus]`. To promote those
claims to T3, they must reproduce on a messy real corpus.

### Prereq (data gate)

- `.agents/memory.md` (any OAW project, or a peer project using OAW) has **≥ 20 entries** accumulated organically (not all written by the same session).
- At least 3 of the entries span the T2/T3 boundary (pre-promotion territory).

### Acceptance criteria (binary)

| Dimension | Pass |
|-----------|------|
| Write discipline (experiment 1 rerun on a **fresh set** of 8 real events) | ≥ 6/6 MUST decisions correct; median body ≤ 60 words |
| Inheritance (experiment 2 rerun, Task A) | ≥ 3/4 exact; 0 hallucinations |
| Promotion (experiment 2 rerun, Task B on real entries) | 0 over-promotes |

If pass: promote the claims in `docs/Memory_System_Experiments_Report_2026-04-21.md`
from T2 → T3.

If fail: investigate which rule was too weak, open follow-up issue.

### Effort + channel

Single future session, ~1–2h. **Not worktree-appropriate** — needs the
accumulated data, which is not something an isolated agent can fabricate.

## M-4 — Track over-refusal rate

### Why

Exp 2's Q3 was a single observation of over-refusal. Named as a failure
mode, but not yet established as a pattern. Need longitudinal data before
treating it as a P1 problem.

### Lightweight instrumentation

Add a convention: when a session observes an over-refusal (agent says
`INSUFFICIENT EVIDENCE` on a question whose answer is in the given files),
log it to `.agents/memory.md` with:

```
## [YYYY-MM-DD] [OVER-REFUSE] <question summary>
File that had the answer: <path>:<line>
Agent's wrong reasoning: <what the agent said>
```

Threshold: **≥ 3 observations in 2 weeks** → promote over-refusal to a P1
fix, revisit `read-discipline.md` wording.

### Effort + channel

- **Convention-adoption**: 0 code change; add `[OVER-REFUSE]` to the memory.md TAG list (this is a S-2-adjacent follow-up; tag itself not yet added).
- **Detection**: manual, in passing. No automation needed until the threshold fires.

Not worktree-appropriate.

## L-5 — `inheritance-check` skill

### Why

Today's Exp 2 was a one-off dogfood. As session count grows, the question
"does the next session's agent actually know what they need to?" recurs.
Build a skill that answers it cheaply.

### Design spec

New file: `templates/.agents/skills/inheritance-check.md`.

**Trigger**: session start, or before merge, or when the user asks "does
the next agent know enough?".

**Parameters**:
- `SOURCES`: files a fresh agent should be able to read (default:
  `AI_CONTEXT.md`, `VERSION.json`, last 5 `.agents/memory.md` entries,
  `ROADMAP.md` first 60 lines — i.e. the Operational lane).
- `QUESTIONS`: 3–5 project-specific questions whose answers are present in
  SOURCES (generated by the current agent, not the user).

**Workflow**:
1. Current agent drafts 3–5 extractable questions + reference answers from SOURCES.
2. Spawn fresh-context subagent. Hand it SOURCES + QUESTIONS. No reference answers.
3. Score: exact match / over-refuse / hallucination.
4. If ≥ 1 over-refuse or ≥ 1 hallucination → flag SOURCES as insufficient,
   propose what to add.

**Acceptance** (for the skill itself):
- Dogfood once: current agent drafts 3 questions from OAW's own project state;
  subagent answers; at least 2/3 exact.
- Skill file passes frontmatter lint.

### Effort + channel

~1–2h. **Worktree spike OK** — it's a doc-only skill, low-risk.

## L-6 — `wiki_sync promote-candidates` subcommand

### Why

Promotion decisions (memory → wiki) are currently implicit. Exp 2 showed
`default STAY when in doubt` is doing its job, but over time a real project
accumulates T3+ entries that genuinely deserve a topic page. Make the
surfacing automatic; keep the decision human.

### Design spec

New `scripts/wiki_sync.py` subcommand: `promote-candidates`.

**CLI surface**:
```
wiki_sync.py promote-candidates
  [--min-tier T3]           # default T3; T2 mode for debugging
  [--memory-file PATH]      # default .agents/memory.md
  [--min-citations 2]       # optional: entry title must appear in ≥ N archive docs
  [--json]                  # machine-readable output
```

**What it does**:
1. Parse `.agents/memory.md` into entries by the `## [DATE] [TAG] Title` header regex.
2. For each entry, extract the tier tag (`[T1]`–`[T5]`).
3. Filter: tier ≥ `--min-tier`.
4. (Optional) For each candidate, grep `docs/archive/` for the entry title
   or a slug-ified variant. Record citation count.
5. Output candidates as a table:

```
| Date       | Tag(s)           | Tier | Citations | Title                   | Suggested wiki topic |
|------------|------------------|:----:|:---------:|-------------------------|----------------------|
| 2026-04-21 | [CLASSIFICATION] |  T3  |    2      | 8 skills pass lint      | Skill_Lint_Baseline  |
```

The **Suggested wiki topic** column is a slug-cased guess; the human makes
the final call.

**What it does NOT do**:
- Does not write to `docs/knowledge/` — output is advisory.
- Does not delete or modify memory entries.
- Does not guess topic pages by LLM; pure string transformation.

### Acceptance criteria

- Subcommand registered in `wiki_sync.py --help`.
- Runs on current branch's `.agents/memory.md` without error; outputs at
  least 1 candidate (the "8 skills pass lint" T3 entry).
- Unit-test case: crafted memory.md with mixed T1/T2/T3/T4 entries →
  `--min-tier T3` returns exactly the T3+ rows.
- `./scripts/harness_check.py` still 8/8 green (no drift).
- Example drift: `example/minimal-project/scripts/wiki_sync.py` synced.

### Effort + channel

acceptance.

## Launch discipline (if you choose to launch any of the above as worktree spikes)

- **One at a time.** Parallel worktree spikes on the same repo create merge
  conflict soup.
- **No push from the spike.** The spike's job is "prove it works in
  isolation". A human opens the PR after review.
- **No destructive actions.** Spec each spike as "implement + test + commit
  locally + report"; never "push or merge".
- **Merge Gate still applies.** Before any spike's branch reaches `main`,
  it must pass harness_check + its own acceptance criteria per
  `CLAUDE.md` §Merge Gate.

## What's expressly NOT in this plan

- **No automation of tier classification**. The `classify-evidence` skill is
  a human (or agent) judgement tool; automating it would defeat its purpose.
- **No LLM-based wiki topic suggestion**. `promote-candidates` is
  deterministic string work; topic naming stays human.
- **No parallel session / shared-state experiments**. That's contingent
  per `Future_Optimization_Plan_Confirmed_2026-04-20.md` §P2-4; don't
  rebuild without 3 people × 2 weeks of real merge pain.
